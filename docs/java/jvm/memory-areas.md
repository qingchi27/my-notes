# Java 内存区域

## JVM 内存结构

JVM 运行时数据区分为**线程共享区域**和**线程私有区域**两大类。

```
JVM 内存
├── 线程共享
│   ├── 堆（Heap）
│   └── 方法区 / 元空间（Metaspace）
└── 线程私有
    ├── 虚拟机栈（VM Stack）
    ├── 本地方法栈（Native Method Stack）
    └── 程序计数器（PC Register）
```

---

## 线程共享区域

### 堆（Heap / GC Heap）

**特点：**

- JVM 中最大的内存区域
- 垃圾回收的主要区域
- 所有线程共享

**存储内容：**

- 对象实例
- 数组

**结构：**

```
堆
├── 新生代（Young Generation）
│   ├── Eden
│   ├── Survivor From
│   └── Survivor To
└── 老年代（Old Generation）
```

**对象分配流程：**

```
new 对象 → Eden
   ↓ Minor GC
Survivor
   ↓ 多次存活
老年代
```

### 方法区 / 元空间（Metaspace）

**存储内容：**

- 类结构信息
- 方法信息
- 字段信息
- 运行时常量池

**JDK 版本变化：**

| 版本 | 实现 |
|------|------|
| JDK 1.7 | 永久代（PermGen） |
| JDK 1.8+ | 元空间（Metaspace） |

**特点：**

- JDK 8 后使用本地内存（Native Memory）
- 不再受 JVM 堆大小限制
- 降低 PermGen OOM 风险

---

## 线程私有区域

### 虚拟机栈（VM Stack）

**存储内容：**

- 局部变量表
- 操作数栈
- 动态链接
- 方法出口

**特点：**

- 线程私有
- 方法执行结束自动释放
- LIFO（后进先出）

### 本地方法栈（Native Method Stack）

为 Native 方法（C/C++）服务，与虚拟机栈类似，也是线程私有。

### 程序计数器（PC Register）

**特点：**

- 记录当前线程执行字节码的行号
- 线程私有
- 唯一不会发生 OOM 的区域

---

## JDK 1.7 vs 1.8

### JDK 1.7

方法区使用**永久代（PermGen）**实现，位于 JVM 堆内。

**存放内容：**

- 类信息
- 方法信息
- 常量池
- 静态变量（JDK 7 开始已移到堆）
- 字符串常量池（JDK 7 开始已移到堆）

**缺点：**

- 大小固定，需通过 `-XX:PermSize`、`-XX:MaxPermSize` 手动配置
- 容易因类加载过多或常量池膨胀导致 OOM

典型异常：

```
java.lang.OutOfMemoryError: PermGen space
```

### JDK 1.8

永久代被**彻底删除**，方法区改为使用**元空间（Metaspace）**实现。

**特点：**

- 使用本地内存（Native Memory），不再占用 JVM 堆
- 默认大小几乎不限（受操作系统物理内存限制）
- 不容易因类加载导致 OOM

常见异常变为：

```
java.lang.OutOfMemoryError: Metaspace
```

**记忆版：**

| 对比项 | JDK 7 | JDK 8 |
|--------|-------|-------|
| 方法区实现 | 永久代（PermGen） | 元空间（Metaspace） |
| 所在位置 | JVM 堆内 | 本地内存 |
| 大小限制 | 固定，易 OOM | 动态扩展，更灵活 |
| 典型 OOM | `PermGen space` | `Metaspace` |

---

## 元空间 vs 永久代

**元空间相对永久代的优势：**

- 不受 JVM 堆大小限制
- 减少 PermGen OOM 发生概率
- 动态扩展能力更强，按需向操作系统申请内存

---

## 堆 vs 栈

| 对比 | 堆 | 栈 |
|------|-----|-----|
| 线程 | 共享 | 私有 |
| 管理 | GC 管理 | 自动释放 |
| 内容 | 对象 | 局部变量、方法调用 |
| 大小 | 大 | 小 |
| 生命周期 | 不确定 | 方法结束即释放 |

---

## 内存溢出 vs 内存泄漏

**内存溢出（OOM）**

内存不够分配新对象，抛出 `OutOfMemoryError`。

**内存泄漏（Memory Leak）**

对象不再使用，但仍被引用，无法被 GC 回收，导致可用内存持续减少。

---

## 内存溢出可能发生的区域

| 区域 | 常见原因 |
|------|----------|
| 堆 | 对象过多、缓存过大 |
| 虚拟机栈 | 线程过多、递归过深 |
| 本地方法栈 | Native 方法调用过多 |
| 方法区 / 元空间 | 类过多、动态代理类过多 |
| 直接内存 | NIO 堆外内存未释放 |

---

## 栈溢出原因

- 递归过深
- 方法调用层级过深
- 栈帧过大
- 线程数量过多（每个线程都有独立栈）

典型异常：`StackOverflowError`

---

## 运行时常量池溢出

- 属于方法区 / 元空间范畴
- 大量字符串或常量未回收导致 OOM
- JDK 7 起 String 常量池已转移到堆中

---

## 对象定位方式

**Java 对象的定位方式**，指 JVM 如何根据对象引用（reference）找到堆中的对象。

目前主要有两种实现方式：

- **句柄（Handle）访问**
- **直接指针（Direct Pointer）访问**（HotSpot 默认采用）

### 对象定位是什么

```java
User user = new User();
```

这里有两个东西：

- `user`：不是对象本身，而是一个**引用（Reference）**
- 真正的对象在 Java 堆里

JVM 必须解决：**`user` 这个引用，如何找到真正的对象？** 这就是对象定位。

---

### 方式一：句柄（Handle）

**结构：**

```
Stack                Handle Pool（句柄池）          Heap
┌──────┐            ┌─────────────────┐
│ user │───────────►│ 对象地址         │──────────► Java 对象
└──────┘            │ Class 地址       │──────────► Class 元数据
                    └─────────────────┘
```

**流程：**

```
引用 → 句柄 → 对象
```

实际上多访问了一层。

**句柄里保存什么？**

一般保存两个指针：

```
Handle
------------------------
instance pointer  →  对象实例
class pointer     →  Class 元数据
------------------------
```

- 访问对象：`reference → handle → instance`
- 访问类信息：`reference → handle → klass`

**GC 移动对象时：**

```
GC 前：Handle → Object A
GC 后：Handle → Object B（引用 user 不用改，只改 Handle）
```

| 优点 | 缺点 |
|------|------|
| 对象移动时只需修改句柄，引用保持稳定 | 每次访问多一次间接寻址，速度较慢 |

---

### 方式二：直接指针（HotSpot 默认）

HotSpot（绝大多数 JVM）采用这种方式。

**结构：**

```
Stack              Heap
┌──────┐           ┌─────────────────┐
│ user │──────────►│ Object          │
└──────┘           │ ├ Mark Word     │
                   │ ├ Klass Pointer │──► Class 元数据
                   │ ├ data          │
                   │ └ Padding       │
                   └─────────────────┘
```

引用直接保存对象地址，例如 `user = 0x12345678`。

- 访问对象：`reference → 对象`
- 获取 Class：`对象 → klass pointer`

**GC 移动对象时：**

```
GC 前：user → 0x001
GC 后：user → 0x888（所有引用都需要更新）
```

现代垃圾回收器（G1、ZGC、Shenandoah）都有高效的引用更新机制，这一成本在实际中可接受。

| 优点 | 缺点 |
|------|------|
| 少一次跳转，访问速度最快 | GC 移动对象时需更新所有引用 |

---

### 两种方式对比

| 对比项 | 句柄 | 直接指针（HotSpot） |
|--------|------|---------------------|
| Reference 指向 | Handle | Object |
| 找对象 | 两次寻址 | 一次寻址 |
| 访问速度 | 较慢 | 更快 |
| GC 移动对象 | 修改 Handle | 修改所有引用 |
| 引用是否稳定 | 稳定 | 不稳定 |
| HotSpot 是否采用 | 否 | 是 |

---

### 为什么 HotSpot 选择直接指针？

Java 程序中最频繁的操作是访问对象字段：

```
句柄方式：user → handle → object → field（多一次寻址）
直接指针：user → object → field
```

**对象访问远比对象移动更频繁**，牺牲少量 GC 更新引用的成本，换取更高的日常访问性能，是更优的整体权衡。

---

### 总结

Java 对象定位主要有两种方式：**句柄访问**和**直接指针访问**。

- **句柄访问**：引用先指向句柄，句柄中保存对象实例地址和类型元数据地址。优点是对象移动时只需修改句柄，引用保持不变；缺点是访问对象需要多一次间接寻址，性能略低。
- **直接指针访问**：引用直接保存对象地址，对象头中的 Klass Pointer 指向类元数据。优点是访问速度快、少一次寻址；缺点是垃圾回收移动对象时需要更新引用。

HotSpot JVM 默认采用**直接指针**方式，因为对象访问的频率远高于垃圾回收移动对象的频率，整体性能更优。

## 变量存储位置

| 变量类型 | 存储位置 |
|----------|----------|
| 局部变量 | 虚拟机栈 |
| 实例变量 | 堆（随对象） |
| static 变量 | 堆（JDK 7 起） |

---

## String 内存

**String 存储位置：**

- String 常量池：JDK 7 起在堆中
- `new String(...)` 创建的对象：在堆中

**`new String("abc")` 过程：**

```
"abc" → 常量池（若不存在则创建）
   ↓
new String → 堆中新建对象
```

因此 `new String("abc")` 可能产生 1～2 个对象。

---

## 对象创建流程

```
类加载检查
   ↓
分配内存
   ↓
初始化零值
   ↓
设置对象头
   ↓
执行构造方法（<init>）
```

**结论：** 默认情况下，对象在堆中创建。具体是否栈上分配见 [垃圾回收与内存分配](./gc-memory.md#对象一定在堆上创建吗)。
